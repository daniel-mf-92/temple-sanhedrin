## Findings

### WARNING: Law 8 - Memory-map Book of Truth API is not attached to the hardware-adjacent mutation sites

Evidence:

- `Kernel/BookOfTruth.HC:2076-2082` defines `BookTruthMemMapRecord`, which encodes a payload and appends it as `BOT_EVENT_NOTE`.
- `Kernel/KExts.HC:131-134` exports `BookTruthMemMapRecord` and `BookTruthMemMapDecode` for callers outside `BookOfTruth.HC`.
- Repository-wide grep at the audited commit finds `BookTruthMemMapRecord` only in its prototype, definition, and extern declaration. There are no call sites in allocator, memory map, page table, or PTE write paths.
- `Kernel/BookOfTruth.HC:2026-2038` truncates virtual and physical page numbers to 18 bits and silently clamps larger values to `0x3FFFF`, so an eventual high-address mapping record would be lossy unless a separate overflow rule is added.

Impact:

The commit creates a memory-map logging interface, but it does not yet record actual memory-map acts at the point where they occur. Law 8 requires page table change logging to be inline with the PTE modification rather than hooked above it, and this commit stops at an exported helper that later code could call from a higher layer. The payload format also risks recording a false-but-valid mapping for addresses outside the 18-bit page range.

Required remediation:

Wire the memory-map record directly into the concrete mapping/free/alias mutation sites, preferably in the same instruction sequence that changes the relevant page table or allocator state. Preserve full address truth or explicitly encode overflow/unsupported ranges instead of clamping them to a valid-looking address.
