## Findings

### WARNING-001: WS13-06 is marked complete while MSR writes are still logged after hardware mutation

Law: Law 8, Book of Truth Immediacy & Hardware Proximity.

The commit changes `MODERNIZATION/MASTER_TASKS.md` from unchecked to checked for `WS13-06 Log all MSR reads/writes`. The implementation surface at this commit still records writes after the raw MSR write has already executed: `SetMSR` calls `RawSetMSR(model_specific_reg,val)` and then calls `BookTruthMSRWriteLog(...)`.

Evidence:

- `MODERNIZATION/MASTER_TASKS.md:239` marks `WS13-06` complete.
- `Kernel/KArchIface.HC:58-62` performs `RawSetMSR(...)` before `BookTruthMSRWriteLog(...)`.
- `Kernel/BookOfTruth.HC:2685-2693` encodes and appends the MSR write note only after the wrapper calls it.

Impact: a fault, reset, SMI side effect, or other interruption between `RawSetMSR` and the append can leave a completed MSR mutation without the synchronous Book of Truth record that the completed WS13-06 line claims exists. This is proximity drift for MSR writes.

Required remediation: move the write record boundary into the same primitive sequence as the MSR write, ideally before or inseparably around the hardware write, with fail-stop behavior if the record cannot be committed.

### WARNING-002: MSR I/O telemetry counters advance before the append result is known

Law: Law 9, Resource Supremacy / Crash on Log Failure.

`BookTruthMSRReadLog` and `BookTruthMSRWriteLog` increment cumulative counters and update `bot_msr_io_last_payload` before returning the result of `BookTruthAppend(...)`. If `BookTruthAppend` returns `FALSE`, the status surface can report cumulative MSR reads/writes and a last payload that were not actually appended to the ledger. Existing MSR wrapper call sites also discard the returned `Bool`.

Evidence:

- `Kernel/BookOfTruth.HC:2679-2682` increments `bot_msr_io_reads` and sets `bot_msr_io_last_payload` before the append result.
- `Kernel/BookOfTruth.HC:2690-2693` increments `bot_msr_io_writes` and sets `bot_msr_io_last_payload` before the append result.
- `Kernel/KArchIface.HC:51-62` calls `BookTruthMSRReadLog(...)` and `BookTruthMSRWriteLog(...)` without checking the returned `Bool`.
- `Kernel/BookOfTruth.HC:3587-3592` reports `cum_reads`, `cum_writes`, and `last_payload` from those pre-append counters.

Impact: the OS can continue after an unrecorded MSR event, and the new status command can make the unsealed attempt look like ledger-backed telemetry. Law 9 requires the OS to halt rather than continue after log write failure.

Required remediation: either fail-stop inside the MSR read/write log path when append fails, or update counters only after a proven successful append and make wrapper callers halt on `FALSE`.
