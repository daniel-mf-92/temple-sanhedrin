### WARNING-001: QEMU fix-queue strict mode can pass while upstream air-gap inputs failed

Applicable laws:
- Law 2, Air-Gap Sanctity
- Law 5, North Star Discipline

Evidence:
- The commit adds `automation/qemu-airgap-fix-queue.py`.
- `build_report(...)` appends gate failures when `qemu-command-manifest-latest.json` or `qemu-airgap-report-latest.json` has `gate_failed: true`.
- `main()` returns non-zero only when `args.strict and report["critical_fix_count"] > args.max_critical`.
- The new smoke explicitly demonstrates that `--strict --max-critical 1` returns success even though its fixture inputs set both source reports to `gate_failed: true`.

Assessment:
This is not a direct air-gap breach because the script is host-only and reads committed/latest report artifacts. It is an enforcement-quality warning: a strict QEMU air-gap fix queue can succeed while its own `gate_failed` field is true because upstream air-gap evidence failed. That weakens Law 2 evidence pressure and makes strict report orchestration harder to trust.

Recommended remediation:
- Make `main()` return `1` for `args.strict and report["gate_failed"]`.
- If the intent is to allow advisory upstream failures, split the fields into `input_gate_failed` and `strict_gate_failed` so the pass/fail semantics are explicit.
