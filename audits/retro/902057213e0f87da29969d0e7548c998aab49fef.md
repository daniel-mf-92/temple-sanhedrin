## Findings

### WARNING: Law 2 host-network fallback is still reachable through the new compiler harness

The new harness does not itself contain network code, but its normal execution path delegates to `qemu-compile-batch.sh`, which delegates to `qemu-compile-test.sh`. That script attempts `curl -sL "$ISO_URL" -o "$ISO_FILE"` when the ISO is missing. This makes a network-dependent host-side test/build path transitively reachable from the new WS10-02 compiler harness. The guest remains protected by `-nic none`, so this is not an observed guest air-gap breach, but it conflicts with Law 2's ban on network-dependent build steps.

Recommended remediation in the builder repo: make the compiler/QEMU path fail closed when the ISO is absent, require a preseeded local ISO by checksum, and remove or quarantine the fetch fallback from any harness used as compliance evidence.

### WARNING: Law 10 readonly image invariant is not verified by the new harness

The harness advertises air-gap enforcement and checks only for `-nic none` in `qemu-compile-test.sh`. The delegated QEMU command uses a writable shared test disk without `readonly=on`. Law 10 specifically treats QEMU launch commands missing `-drive readonly=on` for the OS image as violations. This commit did not introduce the underlying QEMU command, but it created a new top-level compiler harness that relies on that command without extending the guard to cover the readonly invariant.

Recommended remediation in the builder repo: make the harness validate the delegated QEMU command for both `-nic none` and appropriate `readonly=on` coverage, or document that the mutable drive is a separate non-OS transfer disk while the OS image is immutable.
