### WARNING: `BookTruthNSStatus` reuses the MSR baseline payload marker and can misclassify unrelated entries

Evidence:
- `Kernel/BookOfTruth.HC` already defines `BOT_MSR_BASELINE_PAYLOAD_MARKER 0xB9`.
- `BookTruthMSRBaselinePayloadEncode(...)` emits `BOT_MSR_BASELINE_PAYLOAD_MARKER<<56 | (slot&0xFF)<<48 | (value&0x0000FFFFFFFFFFFF)`.
- The new `BookTruthNSStatus(...)` filters with `if ((entry->payload>>56)!=0xB9) continue;`.
- It then treats `entry->payload.u8[0]==1`, `2`, or `3` as boot/key/halt evidence.
- For normal MSR baseline entries, `payload.u8[0]` is the low byte of the MSR value, so any value ending in `1`, `2`, or `3` can be counted as a `BookTruthNSStatus` boot/key/halt hit.

Law impact:
- Law 5 is implicated because this status surface can report false readiness from unrelated MSR baseline records.
- This weakens the audit value of the status command added as evidence for WS13-23 completion.

Recommended remediation:
- Allocate a distinct `BOT_NS_PAYLOAD_MARKER` for the `BookTruthHello`/`botns` event triplet, or decode the payload with a dedicated helper that cannot collide with MSR baseline entries.
- Add a negative fixture containing MSR baseline payloads whose low byte is `1`, `2`, and `3`; the status command should not count them as boot/key/halt.
