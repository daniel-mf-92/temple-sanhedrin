## Findings

### CRITICAL-001: MSR writes mutate hardware before Book-of-Truth append

Applicable laws:
- Law 8: Book of Truth Immediacy & Hardware Proximity

Evidence:
- `Kernel/KArchIface.HC` changes `SetMSR(model_specific_reg,val)` to call `RawSetMSR(model_specific_reg,val);` first, then `BookTruthMSRWriteLog(model_specific_reg,val,BOT_SOURCE_MSR);`.
- `Kernel/BookOfTruth.HC` adds `BookTruthMSRWriteLog`, which composes a payload and appends after the caller has already performed the MSR write.
- LAWS.md requires MSR/TSC reads to avoid wrapper-library indirection and requires the record to be composed and emitted in the same instruction sequence as the hardware event.

Assessment:
For a write, the externally visible hardware state changes before the event has been synchronously recorded. If the subsequent append fails, is bypassed, or is delayed, the system can continue after an unrecorded MSR mutation. This is a direct Law 8 violation for a hardware-control path.

Required remediation:
- Move the fail-stop append decision into the write path before or inseparably around the raw MSR instruction.
- Do not allow the raw MSR write to complete without a guaranteed Book-of-Truth record or an immediate halt.

### CRITICAL-002: MSR log append failure is ignored

Applicable laws:
- Law 9: Resource Supremacy / Crash on Log Failure
- Law 3: Book of Truth Immutability

Evidence:
- `BookTruthMSRWriteLog` returns `BookTruthAppend(...)`, but `SetMSR` ignores the return value.
- `BookTruthMSRReadLog` and `BookTruthMSRWriteLog` return `FALSE` when `!bot_on` instead of halting.
- Law 9 requires continued execution to stop if a log write cannot be completed.

Assessment:
The machine can continue after a failed or disabled MSR log append. For privileged MSR writes, this creates an unledgered control-plane mutation path and violates the fail-stop rule.

Required remediation:
- Treat `BookTruthAppend` failure in MSR read/write logging as fatal in secure-local mode.
- Remove `!bot_on` as a silent skip for hardware-control ledger paths, or make it halt before the raw operation can proceed.

### WARNING-001: MSR payload truncates replay-critical identity

Applicable laws:
- Law 3: Book of Truth Immutability
- Law 8: Book of Truth Immediacy & Hardware Proximity

Evidence:
- `BookTruthMSRIOPayloadEncode` masks `msr` to `0xFFF` and `val` to `0xFFFFFFFFFF`.
- MSR identity and value are not preserved at full width in the compact event payload.

Assessment:
The new log event can prove an MSR access happened, but it cannot reliably replay or audit the full MSR number/value tuple. This weakens the Book-of-Truth record for hardware-proximity evidence.

Required remediation:
- Add a full-width side record, multi-entry encoding, or explicit documented compression rule with collision handling.
- Ensure status/decode tooling can distinguish truncated evidence from complete evidence.

### WARNING-002: The smoke check validates string presence, not fail-stop semantics

Applicable laws:
- Law 5: North Star Discipline
- Law 9: Resource Supremacy / Crash on Log Failure

Evidence:
- `automation/booktruth-msr-io-smoke.sh` checks for wrapper declarations, `RawGetMSR`, `RawSetMSR`, `BookTruthMSRReadLog`, `BookTruthMSRWriteLog`, and `BookTruthAppend(...)` text.
- It does not assert ordering constraints, return-value handling, `bot_on` bypass behavior, or halt-on-append-failure behavior.

Assessment:
The harness is useful as a wiring check, but it cannot catch the two critical Law 8/Law 9 issues above. A commit can pass the smoke script while leaving an unrecorded MSR mutation path.

Required remediation:
- Extend the smoke or static guard to reject `RawSetMSR` before a mandatory logging/fail-stop decision.
- Add a static check that `BookTruthMSR*Log` return values are not ignored on mutating paths.
