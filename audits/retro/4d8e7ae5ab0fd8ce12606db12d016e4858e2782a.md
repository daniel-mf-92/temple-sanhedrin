### WARNING: Law 5 - North Star keypress validation is still synthetic after the `botns(1)` default

Evidence:

- `automation/north-star-e2e.sh:11` changes the default injected command to `botns(1);`.
- At the audited commit, `Kernel/BookOfTruth.HC:81420-81422` defines `botns(Bool halt_after=TRUE)` as `BookTruthHello(FALSE,halt_after)`.
- At the audited commit, `Kernel/BookOfTruth.HC:81396-81406` prints `BoT: keypress=q` without calling `GetChar` when `wait_key` is false.
- `automation/north-star-e2e.sh:91-97` still sends monitor `sendkey q` only after typing the command and pressing return.
- `MODERNIZATION/NORTH_STAR.md:18-21` says the required `BoT: keypress=q` line should occur after the harness feeds `q\n` through the QEMU monitor.

Impact:

The commit makes the default command shorter, but it keeps the North Star harness on the non-blocking synthetic-keypress path. A passing run can therefore prove serial output and guest halt behavior without proving the keyboard input stage required by the North Star. Because the only changed file is the E2E harness, this is not just documentation drift: it directly affects the evidence used to judge North Star progress under Law 5.

Required remediation:

Default the harness to a command path that waits for a real keypress, then feed `q` while the guest is blocked in that wait path. If a non-blocking helper is retained for smoke tests, keep it outside the canonical `automation/north-star-e2e.sh` default or make the script assert that the serial keypress line cannot appear before the monitor `q` is sent.
