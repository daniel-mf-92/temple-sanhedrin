## Findings

### CRITICAL: Law 8 - Mem-map records are emitted after batched mapping changes, not inline with each PTE mutation

Evidence:

- `Kernel/Mem/MemPhysical.HC:117-123` writes a 1 GiB run of PTEs inside a loop, then emits a single `BookTruthMemMapRecord(1,chunk_base,chunk_base,1,1,1)` after the loop.
- `Kernel/Mem/MemPhysical.HC:147-153` writes the uncached alias PTEs for a 4 GiB span inside a loop, then emits one `BookTruthMemMapRecord(3,dev.uncached_alias,0,1,1,1)` after the loop.
- `Kernel/Mem/MemPhysical.HC:100-101` changes the 32-bit device range type to `MRT_UNUSED` before calling `BookTruthMemMapRecord(...)`.
- `LAWS.md` Law 8 requires page table change logging inline with the PTE modification, with the log entry composed in the same instruction sequence as the hardware-adjacent event.

Impact:

The commit improves observability, but it does not meet the "same instruction sequence" requirement for memory mapping changes. A crash or failure inside the mapping loops can leave already-mutated PTEs without corresponding serial records, and the emitted records summarize batches rather than recording each actual PTE change as it occurs.

Required remediation:

Move the record emission into the concrete mutation sequence for each PTE/state change, or add a lower-level helper that performs the PTE write and the Book of Truth append as one indivisible operation. The code should not mark the map/free operation complete before the corresponding log record has synchronously exited the append path.
