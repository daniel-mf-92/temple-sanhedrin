## Findings

### WARNING-001: Disk purge path relies on a name substring keep-filter for Book-of-Truth preservation

Evidence:
- `Kernel/BookOfTruth.HC` adds `BotPathKeep`, which preserves paths only when their names contain `BookTruth`, `BookOfTruth`, or `Sefer`.
- `BotPurgeMask` recursively walks masks and calls `Del(de->full_name,FALSE,del_dir,FALSE)` for any path not accepted by that substring filter.
- `BookTruthReclaimDisk` applies that purge to `blkdev.home_dir`, `blkdev.tmp_filename`'s directory, and global swap/temp masks.

Law impact:
- Law 9 resource supremacy is implicated. The new implementation tries to preserve Book-of-Truth artifacts, but the protection is filename-based rather than tied to a sealed ledger allocation, disk extent, drive, or manifest. A future log path with a different name would be eligible for deletion during reclaim.
- Law 3 immutability is implicated as a drift risk, because accidental deletion of a disk ledger artifact would violate the Book of Truth's no-delete rule even if the serial mirror remains canonical.

Recommended remediation:
- Protect Book-of-Truth disk artifacts by explicit ledger identity or configured allocation, not string containment.
- Add a smoke fixture proving that representative ledger filenames outside the three substring forms are never passed to `Del`.

### WARNING-002: Disk reclaim performs destructive deletes before a durable reclaim summary is appended

Evidence:
- `BookTruthReclaimDisk` performs `Del` calls inside `BotPurgeMask` and directly against `tmp_name`.
- The enclosing `BookTruthReclaimDo` records only one summary payload with `BookTruthAppend(BOT_EVENT_NOTE,BOT_SOURCE_KERNEL,payload)` after the reclaim stages finish.
- If the machine halts or the append fails between the destructive deletes and the summary append, the reclaim intent/count summary is not guaranteed to exist in the ledger.

Law impact:
- Law 8 immediacy/hardware proximity is implicated. The reclaim stage changes disk state before the reclaim event is appended, so the high-level reclaim act is not recorded synchronously with each deletion.
- This is recorded as a warning, not a critical breach, because lower-level disk I/O hooks may still record sector writes when enabled; the gap is the new reclaim API's own act-level evidence.

Recommended remediation:
- Emit a Book-of-Truth event before each purge target or before each bounded purge batch, and fail-stop if the append cannot be written.
- Extend validation to assert ordering: reclaim event append must precede or be inline with each destructive delete path.
