## Findings

### WARNING: Public profile setter can suppress the Book-of-Truth profile event

Evidence:
- `Kernel/BookOfTruth.HC` adds `public U0 BookTruthProfileSet(I64 profile=BOT_PROFILE_SECURE_LOCAL, Bool emit_event=TRUE)`.
- The setter mutates `bot_profile_mode=profile_next` before the optional `BookTruthAppend(...)`.
- `if (emit_event) BookTruthAppend(BOT_EVENT_NOTE,BOT_SOURCE_CLI,payload);` makes the profile transition event skippable by any caller that passes `FALSE`.

Law impact:
- Law 3 forbids code paths that make Book-of-Truth logging skippable.
- Law 8 requires each act to be recorded synchronously and as close to the operation as possible.

Assessment:
- This is not a guest networking breach and does not alter the air-gap.
- It is a policy-audit regression: a security-profile transition can be performed without a corresponding ledger entry.

Recommended remediation:
- Remove the `emit_event` argument from the public setter or make it private/internal-only.
- Log the profile transition inline before or atomically with the state assignment; fail closed if append fails.

### WARNING: Profile payload is encoded as a generic `BOT_EVENT_NOTE`

Evidence:
- `BookTruthProfileSet` and `BookTruthProfileBoot` encode `BOT_PROFILE_PAYLOAD_MARKER` into the payload but emit `BOT_EVENT_NOTE`.
- No profile-specific event type or decode/status helper was added for profile transitions.

Law impact:
- Law 8 immediacy is partially satisfied by direct append on the boot path, but semantic coverage is weak because profile transitions share a generic event type.

Assessment:
- This is auditability drift rather than a direct law breach.
- Sanhedrin and host evidence consumers must pattern-match marker `0xBD` inside `NOTE` payloads to distinguish profile transitions.

Recommended remediation:
- Add a distinct event type such as `BOT_EVENT_PROFILE_TRANSITION`, or add a profile decode helper and coverage check for `BOT_PROFILE_PAYLOAD_MARKER`.
