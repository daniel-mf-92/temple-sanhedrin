## Findings

### WARNING-001: New MM trace hooks mutate allocator state before the Book of Truth record is attempted

Law: Law 8, Book of Truth Immediacy & Hardware Proximity.

`MemPagAlloc` removes or splits a free block, increments `bp->used_u8s`, releases `bp->locked_flags`, restores flags, and only then calls `BookTruthMemMapRecord(1,res,res,res->pags,0xA,1)`. `MemPagFree` similarly changes signatures, counters, and free-list links, then releases the allocator lock and only afterward calls `BookTruthMemMapRecord(2,trace_ptr,trace_ptr,trace_pags,0xB,1)`.

Evidence:

- `Kernel/Mem/MemPag.HC:79-85` records allocation after `bp->used_u8s` is already changed and after `LBtr(...); POPFD`.
- `Kernel/Mem/MemPag.HC:100-121` records free after free-list insertion and after `LBtr(...); POPFD`.

Impact: a crash, halt, interrupt path, or allocator reentry window between the memory-state mutation and the ledger append can leave the Book of Truth missing the exact act it claims to trace. This is a proximity/atomicity drift against the Law 8 requirement that the event and record occur in the same instruction sequence as closely as possible.

Required remediation: move the recording boundary into the mutation path or add a dedicated fail-stop trace primitive that snapshots and commits the Book of Truth entry before the allocator mutation becomes externally visible.

### WARNING-002: New trace append call sites ignore append failure

Law: Law 9, Resource Supremacy / Crash on Log Failure.

`SchedTraceEmit` calls `BookTruthAppend(...)` and discards the returned `Bool`. The new MM call sites also call `BookTruthMemMapRecord(...)`, which returns the `BookTruthAppend(...)` result, but both new callers discard it.

Evidence:

- `Kernel/KTask.HC:153-159` wraps `BookTruthAppend` in a `U0` function with no failure branch.
- `Kernel/KTask.HC:226-227`, `Kernel/KTask.HC:256-257`, and `Kernel/KTask.HC:323-324` invoke the wrapper for scheduler trace events.
- `Kernel/Mem/MemPag.HC:84-85` and `Kernel/Mem/MemPag.HC:121` discard `BookTruthMemMapRecord(...)` return values.
- `Kernel/BookOfTruth.HC:2225-2231` shows `BookTruthMemMapRecord(...)` returns the append result to callers.

Impact: if the append path can return `FALSE`, the scheduler/MM activity continues after a lost trace event. Law 9 requires the OS to die before the log dies, with halt-on-failure behavior in the write path.

Required remediation: make these trace emitters fail-stop on append failure, or prove that `BookTruthAppend` cannot return `FALSE` after init and encode that invariant in the function signature/contract.
