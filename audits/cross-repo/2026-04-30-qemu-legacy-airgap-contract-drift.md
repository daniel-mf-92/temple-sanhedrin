## Invariant Under Audit

Both repos enforce the same Law 2 air-gap outcome, but they currently encode different meanings for the legacy `-net none` fallback:

- TempleOS treats `-net none` as valid when `-nic` is unavailable.
- holyc-inference benchmark tooling treats `-net none` as policy drift and requires `-nic none` in recorded benchmark commands.

This is not a direct air-gap breach. It is a cross-repo contract mismatch that can produce false positives or false negatives when evidence from one repo is consumed by the other.

Relevant law anchors:

- `LAWS.md` Law 2 permits QEMU/VM commands with `-nic none`, with legacy fallback `-net none`.
- The user-supplied hard safety rule for this audit says any QEMU or VM command must explicitly disable networking using `-nic none`, with legacy fallback `-net none`.

## Evidence

- `automation/qemu-airgap-lib.sh:101-110` treats both `-nic none` and `-net none` as disabled-network evidence.
- `automation/qemu-airgap-lib.sh:120-131` requires an explicit no-network option and names both `-nic none` and `-net none` in the error text.

## Findings

### 1. WARNING: Same safe legacy fallback is accepted by TempleOS and rejected by holyc-inference

TempleOS considers `-net none` valid fallback evidence under both code and smoke criteria. holyc-inference marks the same flag as a violation even when it disables networking.

Impact:

- A valid TempleOS fallback run on an older QEMU build would fail holyc benchmark/artifact audits.
- Cross-repo dashboards could report an air-gap failure where the guest was actually air-gapped.
- Builders may "fix" evidence by hiding fallback context instead of recording why fallback was used.

Recommendation:

- Split the holyc status into `airgap_ok` and `preferred_flag_ok`.
- Treat `-net none` as `airgap_ok=true, preferred_flag_ok=false, fallback_reason_required=true` instead of a generic air-gap failure.

### 2. WARNING: holyc-inference rejects mixed `-nic none -net none` evidence more strictly than TempleOS

TempleOS accepts redundant disabled-network flags because both are safe forms. holyc-inference rejects any presence of legacy `-net none` even when `-nic none` is also present.

Impact:

- A command generated by a compatibility wrapper that carries both disabled-network flags would pass TempleOS guardrails and fail holyc artifact checks.
- This weakens cross-repo comparability of historical QEMU evidence.

Recommendation:

- If both flags are present and no network backend/device exists, classify as `airgap_ok=true` with `nonpreferred_legacy_flag=true`.
- Keep rejection for `-net user`, `-netdev`, NIC devices, host forwarding, guest forwarding, and attached netdev devices.

### 3. WARNING: Fallback reason is required by TempleOS criteria but has no holyc artifact field

TempleOS smoke criteria require recording why `-nic none` was not selected when fallback mode is used. holyc artifacts record booleans for explicit `-nic none` and legacy `-net none`, but the inspected benchmark policy does not expose a fallback reason field.

Impact:

- Even if holyc stops treating fallback as an error, it cannot currently preserve the TempleOS-required explanation.
- Historical fallback evidence cannot distinguish "old QEMU lacks `-nic`" from accidental legacy drift.

Recommendation:

- Add a host-side artifact field such as `command_airgap_fallback_reason`.
- Require it only when `legacy_net_none=true` and `explicit_nic_none=false`.

### 4. WARNING: Cross-repo reports can conflate "not preferred" with "not air-gapped"

holyc-inference labels legacy `-net none` as a benchmark-artifact violation in the same list as actual unsafe networking flags. TempleOS distinguishes safe fallback from network-enabling flags.

Impact:

- Law 2 audit output becomes harder to interpret because the same violation channel contains both safety failures and preference drift.
- A real network-enabling flag could be diluted among nonpreferred-but-safe fallback findings.

Recommendation:

- Use separate severities: error for network-enabling options; warning for valid legacy fallback without preferred `-nic none`.
- Preserve a machine-readable field for `network_enabled=false` even when preferred flag policy fails.

### 5. INFO: Both repos reject actual network-enabling options in the inspected paths

Both implementations reject `-nic user` in the direct comparison. The drift is about the safe fallback contract, not about permitting live guest networking.

Impact:

- The TempleOS guest remains air-gapped under the inspected wrappers.
- This report should be handled as evidence-schema alignment, not an emergency networking breach.

Recommendation:

- Keep the strict rejection of `-nic user`, `-net user`, `-netdev`, and NIC devices in both repos.
- Align only the classification of explicit disabled-network fallback evidence.

## Summary

Findings: 5 total.

- Critical: 0
- Warning: 4
- Info: 1

The repos agree on the safety outcome but disagree on how to classify legacy disabled-network evidence. Aligning the schema around `airgap_ok`, `preferred_flag_ok`, and `fallback_reason` would preserve strict air-gap enforcement while avoiding false cross-repo failures for valid fallback runs.
